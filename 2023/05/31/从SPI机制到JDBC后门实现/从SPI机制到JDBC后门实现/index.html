从SPI机制到JDBC后门实现

从SPI机制到JDBC后门实现

1. 前置知识

1.1 Spi机制

Spi机制（Service Provider Interface）服务发现机制，该技术基于classLoader来发现并加载服务。目的是为了将服务接口与服务实现分离以达到解耦，提高服务的扩展性。这个机制简单来说：你只制定接口，不提供实现，接口实现由其他系统应用实现。

一个标准的Spi应该具有以下3部分组成：

  1. Service：一个公开的接口或抽象类，定义一个抽象的功能模块
  2. Service Provider：Service接口的某个实现类
  3. ServiceLoader：Spi机制的核心组件，负责在运行时发现并加载Service Provider

在IDEA编译器中其实使用了很多Spi机制，例如插件就是通过Spi机制实现的。IDEA只提供一个扩展接口，具体由插件开发者自由实现，而IDEA提供了Spi服务发现机制寻找具体插件的实现并对其进行加载，这样就达到了安装插件。在这个情况下，若我们想要安装或卸载某个插件，均不会影响其他代码，实现了可插拔的效果，提高了程序扩展性。

当服务的提供者，提供了服务接口的一种实现之后，在jar包的META-INF/services/目录里同时创建一个以服务接口命名的文件。该文件里就是实现该服务接口的具体实现类。而当外部程序装配这个模块的时候，就能通过该jar包META-INF/services/里的配置文件找到具体的实现类名，并装载实例化，完成模块的注入。通过这个约定，就不需要把服务放在代码中了，通过模块被装配的时候就可以发现服务类了。具体Spi使用可参考：https://www.jb51.net/article/251767.htm。由于篇幅原因不进行过多展示，可自行参考学习。

跟进`loadInitialDrivers()`​方法中，586行体现了Spi机制是如何破坏Java的双亲委派模型的（但是这并不是重点），在602行调用`driversIterator.hasNext()`来判断是否有下一个JDBC的驱动程序需要加载和注册，实现流程如下所示。
  1. java.util.Iterator#hasNext：在587行生成迭代器后602行调用其hasNext()​方法，跟进后发现在这个方法中出现了两个变量knownProviders​和lookupIterator​，其中前者是已知服务提供者的迭代器，它存储了在之前的迭代中已经找到的所有服务提供者，包括已经实例化的和还未被实例化的；而后者是服务提供者查找器的迭代器，它用于查找新的服务提供者。所以在 hasNext()​方法中，先判断 knownProviders​ 中是否还有下一个服务提供者，如果有则直接返回 true，否则再通过 lookupIterator​查找新的服务提供者，如果找到则添加到 knownProviders​ 中，最终返回 true。
  2. java.util.ServiceLoader.LazyIterator#hasNext：acc​是一个 AccessControlContext​ 类型的变量，用于实现 Java 安全模型中的权限控制。Java 安全模型中包含了许多安全策略和权限控制的机制，其中 AccessControlContext​ 用于描述当前线程的访问控制上下文，包括当前线程的访问控制器、权限集等信息。在 hasNext()​ 方法中，如果 acc​ 为空（即没有指定安全策略），则直接调用 hasNextService()​ 方法来判断是否有下一个服务提供者；如果 acc​ 不为空，则使用 AccessController.doPrivileged()​ 方法来执行 hasNextService()​ 方法，以此来保证在访问服务提供者时符合安全策略的要求。
  3. java.util.ServiceLoader.LazyIterator#hasNextService：如果nextName​（存储下一个服务提供者的名称）不为 null，说明已经找到下一个服务提供者，直接返回 true。如果configs​（存储服务提供者的配置文件的枚举对象）为 null，说明还没有加载服务提供者的配置文件，需要先加载配置文件。加载配置文件的过程中，会根据服务提供者的名称查找对应的配置文件，并将其存储在configs​变量中；而在这里配置文件的名称默认是在META-INF/services​下的，这也就是为什么Spi的配置文件要写在这目录下。如果已经加载了配置文件，但是没有找到下一个服务提供者，则需要遍历已经加载的配置文件，查找下一个服务提供者。具体来说，会先判断 pending​ （用于存储当前服务提供者的所有实现类的名称）变量是否为 null 或者 pending​ 迭代器中是否还有元素。如果 pending​ 不为 null，且pending​迭代器中还有元素，则说明已经找到下一个服务提供者，将其存储在nextName​变量中，并返回 true。否则，需要继续遍历已经加载的配置文件，查找下一个服务提供者。如果已经遍历完所有的配置文件，仍然没有找到下一个服务提供者，则返回 false，表示不存在下一个服务提供者。总的来说，该方法的作用是判断是否存在下一个服务提供者，并将其存储在nextName​变量中，以便在下一次调用 next() 方法时使用。具体实现逻辑比较复杂，需要依次加载配置文件、解析配置文件，并遍历已经解析的配置文件才能找到下一个服务提供者。

​​​​​​​​​​

其次判断通过后进`driversIterator.next()`​的，由于1-2步和前面一模一样，这里就不过多去阐述，解释一下最后一步即可。
  1. 首先代码调用了hasNextService()​查看是否存在下一个服务提供者，由于我们在hasNext()​上面的流程中已经寻找到并且将nextName​赋值，所以可以直接return true；​随后获取下一个服务提供者的类名，并将 nextName​ 置为 null​，以便于下次循环调用hasNext()​时走完整个流程。之后通过反射获取服务提供者类 c​，并将其加载到内存中，并检查该服务提供者是否实现了指定的服务接口 service​，如果不是则抛出异常。调用 newInstance()​ 方法实例化服务提供者 p​，并将其转换为指定的服务类型 S​。将服务提供者实例 p​ 存储到 providers​ 集合中，以备后续使用。而后返回服务提供者实例 p​。总之，这段代码的作用是从配置文件中加载服务提供者并实例化服务。

​​

那么就可以解释刚刚到问题，为什么第一步加载驱动可以省略还能正常运行，就是因为Java里的数据库驱动使用了Spi机制，会默认的寻找classpath下所有的`META-INF/services`​目录里文件名为`java.sql.Driver`​里文件内容找出来并进行加载，所以就不需要我们手动去加载驱动了。​

3. 后话

如果当我们获取了目标服务器权限后，做以上的事情就会比较轻松（毕竟维持后门），但是在攻击中是否可以利用这个方式进行攻击？我觉得也是可以的。在Springboot中由于默认不解析脚本文件了，使得文件上传的点会比较难以利用。虽然网络上也有很多已知点攻击利用方式，例如ssh，定时任务等，但是这些对于有系统检测到应用系统来说很容易告警，这不得不考虑也可以成为一种攻击思路。

虽然springboot大多都是以maven进行管理第三方组件了，但并不是所有的组件都可以在中央仓库上找到，还有很多个人自己开发的等，此时就会把自己开发的jar包放到某个目录下作为项目的classpath，也就是idea里的lib，那么此时该目录下的所有jar文件都会成为第三方组件库，如下所示。
